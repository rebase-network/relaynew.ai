## 本轮确认的过时点

### 1. 数据库文档落后于当前迁移状态

发现：

- `docs/DATABASE_SCHEMA.md` 仍以“initial schema”口径描述数据库，但当前可执行 schema 已经包含后续迁移。
- `submissions` 缺少 `description` 字段说明。
- `probe_credentials` 没有写清楚提交审核流与自助测试流的 key 持久化边界。
- `probe_region` 已落库，但当前 public contract 仍只开放 `global`。

影响：

- 容易让后续开发者误判哪些字段已经可用。
- 也会让运营 / 审核流程对测试 key 的保存边界理解不一致。

### 2. 前端部署文档对 GitHub Workers Builds 的约束不够强

发现：

- `docs/DEPLOYMENT.md` 与 `docs/CLOUDFLARE_WORKERS_BUILDS.md` 里，部分语气仍像“推荐路径”。
- 某些描述会让人误以为可以通过 dashboard 手动触发或本地 `wrangler deploy` 发布 `web` / `admin`。

影响：

- 与仓库当前部署规则冲突：
  - `relaynew.ai`
  - `admin.relaynew.ai`
  必须且只能通过 GitHub 触发的 Cloudflare Workers Builds 发布。

### 3. Internal API 备注没有完全跟上路由拆分

发现：

- `POST /public/submissions` 已经在 public 路由模块中实现，但文档没有把模块归属写清楚。
- `POST /public/probe/check` 的“单独路由、默认不持久化 key”边界也不够直白。
- 当前仓库还没有真正暴露 `/internal/*` HTTP 面，但文档没明确说明这一点。

影响：

- 容易让后续继续新增接口时把 public、admin、internal 的边界再次写混。

## 已完成的文档修正

本轮已经完成以下同步更新：

- `docs/DATABASE_SCHEMA.md`

## 结论

本轮没有发现新的架构级偏差，主要是“实现已经前进，但文档描述还停留在上一阶段”的问题。

这类偏差虽然不会立刻导致线上故障，但会直接影响：

- 中国运营团队接手时对流程边界的理解
- 后续继续拆分 public / admin / internal 时的判断
- 部署时是否误走错误路径

因此本轮文档修正应视为必要收口，而不是可选整理。
