# 2026-04-20 提交站点页面修复方案

## 背景

当前 `提交站点` 页面已经能完成基本提交流程，但桌面端的版式和信息层级存在明显问题：

- 左侧 hero 会被右侧表单强行拉伸，底部出现大块无意义留白
- hero 标题在窄列中被切成过多短行，首屏观感不稳定
- hero 底部三段说明被挤压为小号脚注，不像清晰的信息模块
- 表单字段标签大量复用 `mono + uppercase` 风格，不适合中文表单阅读
- 价格表仍然偏散，列对齐与行扫描效率不够高
- `测试API Key` 被放在底部提交条里，视觉上像附加说明，不像核心必填字段

本轮目标不是改业务逻辑，而是先把 `提交站点` 页面重新收成一个更稳定、更好读、更适合中文用户的桌面端表单页。

## 修复目标

- 修复桌面端 hero 与表单的结构失衡问题
- 让首屏先呈现明确的主引导，再进入信息填写
- 优化中文标题、说明和字段标签的阅读体验
- 让价格表更接近真正的紧凑表格，而不是多层小卡片
- 让 `测试API Key` 回到主表单信息流中，避免被误读为次要项
- 保持当前提交接口、字段结构、Playwright 提交流程不变

## 设计原则

### 1. 先修结构，再修细节

当前最影响观感的是布局结构而不是颜色或单个字号，因此优先级应为：

1. 根布局比例与拉伸行为
2. hero 信息组织
3. 表单字段层级
4. 价格表收敛
5. 提交区与反馈区细调

### 2. 保持中文优先

- 中文标签不再默认使用过强的英文式 `uppercase mono`
- 中文标题行数尽量控制在 2-3 段，不做碎裂式断行
- 说明文案只保留能帮助用户填写的内容，不堆叠口号式说明

### 3. 表单页优先“可填写”，不是“像宣传页”

- hero 需要提供引导，但不能占据过高视觉权重
- 表单区需要更稳定、更清晰的阅读节奏
- 重要字段应该出现在用户自然的填写路径中，而不是放到页脚

## 拟实施方案

## 一、页面整体布局

### 现状问题

- `apps/web/src/pages/submit-page.tsx` 当前用两列 `grid`
- 左右列默认拉伸到同高，导致左侧 hero 被右侧表单撑高
- 左列宽度偏窄，标题换行碎裂

### 调整方案

- 保留桌面端双列结构，但取消左列默认拉伸
- 让 hero `self-start`
- 重新调整桌面端列宽比例，给 hero 更合理的标题行宽
- 如果调整后首屏仍偏高，则适当压缩 hero 内部垂直间距

### 预期结果

- 左侧不再被右侧表单拉长
- hero 高度由自身内容决定
- 首屏更像“引导 + 表单”，而不是“两块等高海报”

## 二、Hero 区重排

### 现状问题

- 标题太长、列太窄，导致断行过碎
- 三段说明在桌面端被放成三列小字，阅读效率低
- hero 底部信息像脚注，不像辅助引导

### 调整方案

- 收紧 hero 标题字号和最大宽度关系，让标题回到更自然的中文断行
- 精简 hero 区的层级：保留标题、摘要、三条流程提示
- 将“先审核 / 整理信息 / 初始测试”改成更紧凑的说明结构：
  - 优先考虑纵向列表
  - 如桌面端需要横向组织，则使用更短的标题和更强的对齐
- 避免让三条说明占据 hero 最底部的大面积空白

### 预期结果

- hero 更像简洁清晰的提交引导
- 中文标题不会被切碎
- 用户一眼能理解“提交 -> 审核 -> 测试 -> 收录”的流程

## 三、表单字段层级调整

### 现状问题

- 中文字段标签套用全局 `form-field` 风格后显得过碎
- `基础信息 / 站点介绍 / 支持模型及价格表` 的区块头还不够明确

### 调整方案

- 为提交页补一套更适合中文的字段标签样式：
  - 保留轻量层级感
  - 减弱 mono/uppercase 的侵略性
- 让 section 标题更像表单章节，而不是轻量提示条
- 压缩区块间距，提升信息密度，但保留足够的输入留白

### 预期结果

- 表单阅读更顺
- 用户能快速识别当前填写的是哪一组信息

## 四、价格表重构

### 现状问题

- 每行仍然像独立小卡片，扫描效率不高
- 表头存在感弱
- placeholder 过淡、过小，辅助作用不足
- 操作列显得松散

### 调整方案

- 桌面端把价格表进一步向“紧凑表格”收敛：
  - 强化表头与列对齐
  - 收紧每行 padding
  - 降低每行卡片感
- 保留移动端逐行填写模式，但桌面端尽量接近真正的表格输入
- 调整 placeholder 样式，让模型示例与价格示例更容易读
- 删除/清空按钮位置和尺寸同步收敛，避免操作列太跳

### 预期结果

- 桌面端更容易连续填写多行模型价格
- 价格表的“表格感”比现在更强

## 五、测试 API Key 区域回归主信息流

### 现状问题

- `测试API Key` 目前在底部提交条里
- 用户容易把它误读成提交附带设置，而不是必填核心字段

### 调整方案

- 将 `测试API Key` 从页脚提交条移回主表单区
- 放在更符合心智的位置：
  - 方案 A：放在基础信息之后
  - 方案 B：放在价格表之后，但作为单独 section
- 底部提交区只保留：
  - 提交按钮
  - 极简结果提醒 / 风险提醒

### 当前推荐

- 优先采用方案 B：放在价格表之后作为单独 section

原因：

- 用户通常先填写站点信息和价格信息，再填写专门用于测试的 Key
- 逻辑顺序更自然
- 页面底部提交区会更干净

## 六、提交区与反馈区收敛

### 调整方案

- 底部提交条只承担最终动作，不再塞入主要字段
- 提交成功后的反馈卡继续保留，但信息层级再收紧
- 错误提示与字段错误位置保持现有逻辑，不改交互规则

## 实施顺序

1. 调整 `submit-page` 根布局
2. 重排 `submit-hero`
3. 给 `submit-form` 增加更合适的中文字段层级
4. 重构价格表样式
5. 移动 `测试API Key` 到主信息流
6. 收敛底部提交条与反馈区
7. 运行本地类型检查、构建与 `public` Playwright

## 预计修改文件

- `apps/web/src/pages/submit-page.tsx`
- `apps/web/src/features/submit/submit-hero.tsx`
- `apps/web/src/features/submit/submit-form.tsx`
- `apps/web/src/styles.css`
- 如有必要：`e2e/public.spec.ts`

## 验证方式

- `corepack pnpm --filter @relaynews/web run typecheck`
- `corepack pnpm --filter @relaynews/web run build`
- `corepack pnpm exec playwright test e2e/public.spec.ts`
- 人工检查：
  - 桌面端 `/submit`
  - 移动端 `/submit`
  - 多行价格表填写场景
  - 提交成功和字段错误场景

## 验收标准

- 桌面端左 hero 不再被表单撑高
- hero 标题断行明显改善
- 三条流程说明不再像页脚小字
- 中文表单标签更清晰、更好读
- 价格表桌面端更紧凑、列感更强
- `测试API Key` 回到主信息流
- 提交流程与现有测试通过
